# Lesson 0003: Agent 的工具调用循环

这一课只做一件事：

**把“工具调用循环”从一句抽象定义，变成你脑子里能放电影的一个具体过程。**

如果你现在觉得这课空洞，很正常。因为 `tool_call`、`messages`、`tool_result` 这些词都很像“名词”，但真正难的是看见它们在一次执行里怎么流动。

所以这次我们不先背定义，先看一轮完整执行。

## 先看结论

Agent 的工具调用，本质上不是：

```text
模型自己去调用 Python 函数
```

而是：

```text
模型提出请求 -> 你的 Python 代码执行 -> 执行结果再回给模型
```

你可以把模型当成“会做决定的脑子”，把应用代码当成“真正干活的手脚”。

## 先用一个现实类比

把整个过程想成你在饭店点菜：

- 你是用户：说出目标
- 服务员是模型：判断该下什么单
- 后厨是 Python 工具函数：真正做菜
- 前台系统是应用代码：负责传单、收结果、继续推进

关键点在这里：

**服务员不会自己冲进后厨做菜。**

服务员只会说：

```text
这桌要一份番茄炒蛋
```

后厨做完之后，服务员再把菜端回来。

Agent 里的工具调用也是一样：

- 模型不会直接执行函数
- 模型只会提出“我想调用某个工具”
- 真正执行的是你的 Python 代码

## 一轮完整执行，到底发生了什么

假设用户输入：

```text
计算 23 * 19，然后解释结果
```

现在我们一步一步走。

### 第 1 步：程序先准备消息

```python
messages = [
    {"role": "system", "content": "你是一个学习助手，需要计算时优先使用 calculator"},
    {"role": "user", "content": "计算 23 * 19，然后解释结果"},
]
```

这时 `messages` 里只有两件事：

- 系统规则
- 用户任务

### 第 2 步：程序把 messages 和 tools 发给模型

```python
response = client.chat.completions.create(
    model=model,
    messages=messages,
    tools=TOOLS,
)
```

这里最重要的不是语法，而是含义：

- `messages` 告诉模型“现在发生了什么”
- `tools` 告诉模型“你有哪些外部能力可用”

就像你对服务员说：

```text
这是顾客的要求，这是餐厅菜单
```

### 第 3 步：模型不一定直接回答

如果模型觉得这题需要更可靠地计算，它可能返回的不是最终答案，而是类似这样一个请求：

```json
{
  "name": "calculator",
  "arguments": {
    "expression": "23 * 19"
  }
}
```

这不是“结果是 437”。

这是：

```text
请帮我调用 calculator，参数是 23 * 19
```

这一步是很多人第一次学时最容易混淆的地方。

## 最重要的分界线

请牢牢记住这两句话：

```text
tool_call = 模型发出的请求
tool_result = 你的代码执行后的结果
```

两者完全不是一回事。

### tool_call 是什么

像工单，像点菜单，像函数调用申请。

### tool_result 是什么

像后厨做完菜后给出的成品，像函数真正执行后的返回值。

## 第 4 步：Python 代码读取这个请求

应用代码会检查：

```python
if message.tool_calls:
```

意思不是“工具已经执行完了”，而是：

```text
模型刚才请求调用工具了
```

然后代码会做这几件事：

1. 取出工具名
2. 解析参数
3. 找到对应 Python 函数
4. 执行函数

例如：

```python
tool_name = tool_call.function.name
arguments = json.loads(tool_call.function.arguments)
result = calculator(arguments["expression"])
```

真正算出 `437` 的，是这里的 `calculator(...)`，不是模型。

## 第 5 步：为什么要把工具结果再塞回 messages

这是第二个最容易觉得抽象的地方。

执行完工具后，代码通常会追加两类内容：

```python
messages.append(模型刚才发出的 tool_call)
messages.append(工具执行后的 tool_result)
```

很多人会问：

```text
已经算完了，为什么不直接 print 结果？
```

因为用户的要求不是“只要一个数字”，而是：

```text
计算 23 * 19，然后解释结果
```

工具只能负责“算”。
模型还要基于算出来的结果，生成自然语言解释。

所以模型必须知道：

- 自己刚才请求了什么工具
- 工具返回了什么结果

否则它下一轮根本不知道发生了什么。

## 把 messages 想成“任务现场记录”

不要把 `messages` 只理解成“聊天记录”。

在 Agent 里，它更像一份持续更新的任务现场记录。

一次工具调用后，`messages` 大概会从这样：

```python
[
  {"role": "system", "content": "..."},
  {"role": "user", "content": "计算 23 * 19，然后解释结果"}
]
```

变成这样：

```python
[
  {"role": "system", "content": "..."},
  {"role": "user", "content": "计算 23 * 19，然后解释结果"},
  {"role": "assistant", "tool_calls": [{"name": "calculator", "arguments": "{\"expression\":\"23 * 19\"}"}]},
  {"role": "tool", "content": "437"}
]
```

你可以这样理解：

- `user`：老板提要求
- `assistant tool_call`：模型说“我要用 calculator”
- `tool`：代码说“算完了，结果是 437”

这样模型第二轮再看到这些消息时，才能接着说：

```text
23 * 19 = 437。这个结果表示...
```

## 第 6 步：为什么需要 while 循环

因为模型不一定一次就结束。

有三种可能：

1. 不需要工具，直接回答
2. 调用一次工具，再回答
3. 调用多个工具，再回答

所以程序不能写成“只请求一次模型就结束”，而要写成：

```python
while True:
    问模型下一步
    如果模型要工具:
        执行工具
        把结果回给模型
        继续循环
    否则:
        输出最终答案
        break
```

循环的意义不是“高级”，而是为了支持：

```text
看结果 -> 再决定下一步
```

这正是 Agent 和普通 Chatbot 的分水岭。

## 和 simple_chat.py 的区别，到底差在哪

`simple_chat.py` 大概是这样的：

```text
用户输入 -> 模型直接回答 -> 程序打印
```

它没有中间过程。

工具型 Agent 则变成：

```text
用户输入
-> 模型决定是否调用工具
-> 代码执行工具
-> 工具结果回给模型
-> 模型再组织最终回答
```

真正多出来的，不是“能聊天”，而是：

**模型可以根据当前状态决定下一步行动。**

## 这一课你至少要咬住的 3 个判断

如果你暂时记不住全部代码，至少先稳住这 3 个判断：

1. 模型不会直接执行工具，它只会提出工具调用请求。
2. `messages` 不只是聊天历史，它还是任务状态。
3. `while` 循环是为了让模型能根据工具结果继续决策。

## 反过来判断：哪些情况需要工具

试着先自己答，再看下面的解释。

### 问题 1

```text
什么是 Python 的 list？
```

通常不需要工具。
因为这是模型已有知识就能直接回答的问题。

### 问题 2

```text
读取 notes.md，总结我上次学到了什么
```

需要工具。
因为模型默认看不到你的本地文件，必须通过文件读取工具拿内容。

### 问题 3

```text
计算 128 * 37，再用一句话解释
```

通常适合工具。
因为计算结果可以交给 `calculator`，比模型心算更稳。

### 问题 4

```text
把我的学习目标保存到本地文件
```

需要工具。
而且这是有副作用的工具，需要代码层做权限和安全控制。

## 一个你可以自己复述的最小故事

如果你能自己顺着说出下面这段话，说明你已经开始真正理解了：

```text
用户先给出任务。
程序把任务和工具说明发给模型。
模型如果觉得要用工具，就返回一个 tool_call 请求。
Python 代码读取这个请求，执行真实函数，得到 tool_result。
程序把 tool_result 放回 messages。
模型看到结果后，再决定给出最终回答还是继续调用工具。
```

这段话比背定义更重要。
